Plotting cloud fraction retrievals with python, cartopy, and xarray

Posted on 17-11-2017 | In Earth Observation

In this post, I will show how to plot cloud fraction retrieval over maps, using python, cartopy, and the wonderful xarray library.
The data I am going to use has been generated by early radiance measurementes collected by the TROPOMI instrument, on board of the freshly launched ESA‘s Sentinel-5P satellite, processed by KNMI‘s FRESCO algorithm.
The workflow can be easily extended to a broad range of netCDF4-based earth observation measurements.

I will start by importing the needed libraries:

Reading the data

I am now going to collect my input files. Sentinel-5P maps the atmosphere around the entire planet every day/fifteen orbits. I thus transferred all the products from a day of sampling from the Sentinel-5P processing facility’s ftp server to my work MacBook. That is some 2.6GB of data.
Each of the FRESCO products is stored in an dedicate sub-directory of a parent node containing all generated products for that particular day. Instead of copying each single file by hand, I transferred the entire sub-tree and let python’s glob do the work for me. In this case, iglob is going to recursively scan the specified directory and return a generator, which I then cast to a list, of all found netCDF4 files.

Number of input files: 15

I now have a bunch of netCDF4 files, containing global measurements of several atmospheric properties, how do I even import them in python?
The xarray library comes to help. xarray can seemlessly read a netCDF4 file and return a Dataset structure, which is is xarray’s multi-dimensional equivalent of a pandas‘s DataFrame.

What’s happening here? Where are the data? The returned Dataset looks disheartening empty, except for an admittedly quite comprehensive set of metadata.
As it turns out (I listed the file contents with h5ls to find out), the data itself is fully confined into the PRODUCT group of the original netCDF4 file, which is not immediately clear by just opening the data set with xarray.
Luckily, xarray allows to specify a path to the netCDF4 group of interested, in the open_dataset function:

The cloud fraction retrieval is stored as an array of 2526272 floats, measured along 5639 scanlines (along-track direction) and 448 ground pixels (across-track direction). Each of the pixels in each of the scanlines is associated with a latitude/longitude tuple, identifying the centre-pixel position.
Let’s have a look at the latitude and longitude coordinates:

Now that is a fancy plot! What happened here is that xarray plotted both the latitude and the longitude coordinates as function of both ground pixel and scanline. If I fix one of those indexes, I should be able to obtain a more orthodox plot. I am going to choose the middle pixel of each scanline and track his position along all scanlines.

Does it look like clouds? Not quite yet.
I will now try to plot the cloud fraction retrieval on a map, using cartopy‘s equirectangular projection (plate carrée) and matplotlib‘s pcolormesh function. That is going to take into account all ground pixels for all scanlines in our data set and plot them on the map with a colour depending the cloud fraction value. I defined my coordinates to be the latitude and longitude. I won’t have to define a meshgrid for my values as xarray is going to do it for me.

Woah, what’s going on here? This is not exaclty what I had expected. There seems to be some spurious horizontal lines across the entire plot. According to this very detailed answer on stackoverflow, I found out that cartopy can’t properly pcolormesh geodetic coordinates yet. The workaround is to pcolormesh PlateCarree coordinates. This can be achieved by converting the longitude values to valid PlateCarree topology by adding 360 to the values below 0.

What about plotting all the fifteen input files on the same map, to get our global cloud fraction retrieval?
I see at least two ways how we could accomplish this taks:

  1. Plot each data set individually on the same map;
  2. Combine all data sets into one global data set, and plotting this global data set instead.

As it turns out, solution 1. comes with all kind of issues related to multiple colorbars: matplotlib will show one colorbar for each of the sub-plots, resulting in a long array of color bars shown beside our map. Which is, if you think about it, a very sensible approach, as the values for the different plots could fall into different ranges. There are ways to deal with this problem, but I found it quicker to go with solution 2. In addition, I can make use of xarray capabilities to concatenate two DataArray, and that’s going to make me look cool, I guess.

Let me describe my approach: I am going to loop over the remaining input files, read them one by one into an xarray‘s DataArray, recondition the scanline index to be contiguous with the previously read values, apply the workaround on the longitude values, and concatenate the new data set with the previously read ones along the scanline dimension. It sounds like a lot of work, but xarray is here to make our lives simpler.
